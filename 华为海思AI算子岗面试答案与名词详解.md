# 1. 岗位职责逐词详解

## 1.1 算子（Operator）

算子是计算图中的一个基本计算单元。输入若干张量，按照确定的数学规则产生输出张量，例如 MatMul、Conv、Softmax、LayerNorm、RMSNorm、Attention、TopK、Gather。

一个工业级算子通常包含：

- **数学语义**：输入、输出、公式、广播规则、数据类型和异常行为。
- **Shape 推导**：根据输入形状推出输出形状，处理动态 Shape。
- **Kernel**：真正运行在 CPU、GPU、NPU 上的并行代码。
- **Host 侧逻辑**：参数检查、Tiling、Kernel 选择、Launch 配置。
- **正确性测试**：与高精度参考实现对齐。
- **性能测试**：覆盖典型 Shape、数据类型和硬件型号。

面试回答可以概括为：

> 算子开发不是把公式翻译成代码，而是同时保证语义正确、数值稳定、Shape/类型完整，并根据硬件的计算单元和存储层次设计并行切分与数据搬运，使端到端性能真正提升。

## 1.2 大模型训练算子

训练不仅有前向计算，还要保存中间激活、进行反向传播并更新参数。训练算子重点关注：

- 前向和反向 Kernel 是否高效。
- 激活、梯度、优化器状态的显存占用。
- FP16/BF16 混合精度和 FP32 累加。
- 梯度检查点、重计算、梯度累积。
- 数据并行和模型并行中的通信。
- 训练稳定性与确定性。

典型训练热点包括 GEMM、Attention、归一化、激活函数、Embedding、优化器、通信算子。

## 1.3 大模型推理算子

推理没有梯度和优化器状态，但需要关注：

- 首 Token 时延（TTFT）。
- Token 间时延（TPOT / ITL）。
- 吞吐量（tokens/s、requests/s）。
- KV Cache 容量与访问效率。
- 动态批处理和请求调度。
- Prefill 与 Decode 工作负载差异。
- 量化、算子融合和低精度计算。

Prefill 更像大矩阵计算，通常计算密集；Decode 每步只生成少量 Token，需要反复读取权重和 KV Cache，往往更受显存带宽和调度开销限制。

## 1.4 生成式推荐

生成式推荐使用生成模型直接产生候选物品、序列或推荐解释，而不只是传统的“召回—排序”。常见计算包含：

- 用户和物品 Embedding。
- 长序列 Attention。
- 自回归生成。
- 大规模词表或物品表上的 Softmax / TopK。
- 多模态编码与融合。

算子优化角度重点是长序列、稀疏访问、Embedding 查表、TopK、动态 Shape 和批处理。

## 1.5 数据合成

数据合成是使用规则、仿真器或生成模型产生训练数据。大模型场景包括指令生成、知识蒸馏、难例生成、数据增强和偏好数据生成。

岗位中的价值不是“写 Prompt”这么简单，而是高吞吐地运行生成模型、过滤和打分数据，并保证数据质量、去重、隐私与分布合理。

## 1.6 算子加速

算子加速的目标是降低时延、提高吞吐或减少资源占用。常见路径：

1. **先测量**：明确端到端瓶颈和热点算子。
2. **建模型**：计算 FLOPs、访存字节、算术强度和理论上限。
3. **改数据流**：减少 HBM/DDR 访问，提高片上复用。
4. **改并行映射**：提高并行度和负载均衡。
5. **用专用单元**：矩阵计算单元、向量单元、Tensor Core/NPU Cube。
6. **做融合**：避免中间张量落到外部内存。
7. **做流水**：重叠搬运和计算。
8. **回归验证**：验证精度、各种 Shape 和端到端收益。

## 1.7 数学等价变换

在保持数学语义或允许误差范围不变的前提下，把表达式改写为更适合硬件的形式。例如：

- `log(softmax(x))` 改为数值稳定的 `logsumexp` 形式。
- 多个线性层合并为一次更大的 GEMM。
- 推理阶段将 BatchNorm 参数折叠进 Conv。
- `x * sigmoid(x)` 识别为 SiLU，使用融合实现。
- Attention 通过分块在线 Softmax 避免显式物化完整注意力矩阵。

要强调：浮点数不满足严格结合律，因此“实数域数学等价”不一定“逐位相同”。必须重新做精度验证。

## 1.8 精度分析

精度分析研究优化前后结果为什么不同、是否可接受。通常包括：

- 选择 FP64/FP32 参考实现。
- 比较绝对误差、相对误差、ULP、余弦相似度。
- 分析误差随 Shape、数据分布、归约长度的变化。
- 检查 NaN、Inf、溢出、下溢和消减误差。
- 定位首个发生显著误差的中间张量。
- 评估算子误差是否影响最终模型指标。

常用判据：

```text
absolute_error = |y - y_ref|
relative_error = |y - y_ref| / max(|y_ref|, epsilon)
```

归约类算子应特别关注累加顺序和累加精度；Softmax、Exp、Log 应特别关注范围与数值稳定性。

## 1.9 计算图优化

计算图是由算子节点和张量边组成的有向图。图优化包括：

- 常量折叠。
- 死代码和无用算子消除。
- 公共子表达式消除。
- 算子融合。
- Layout 传播和格式转换消除。
- 内存复用、生命周期分析。
- 静态 Shape 特化。
- 子图替换和高性能 Kernel 选择。

图优化关注全局，而单算子优化关注局部。一个单算子快 20%，如果它只占端到端 1%，收益很小；一次图融合可能同时减少多个 Launch 和中间张量搬运，端到端收益反而更大。

## 1.10 流水编排（Pipeline）

流水编排是把数据搬运、计算、通信拆成阶段并重叠执行。例如双缓冲：

```text
时间 t0：搬运 Tile 0
时间 t1：计算 Tile 0，同时搬运 Tile 1
时间 t2：计算 Tile 1，同时搬运 Tile 2
```

理想情况下总时间从 `T_copy + T_compute` 接近 `max(T_copy, T_compute)`，但还受同步、缓冲区数量和资源占用限制。

## 1.11 深度融合

融合把多个相邻算子放进同一 Kernel。例如：

- Bias + Activation。
- MatMul + Bias + GELU。
- Add + LayerNorm。
- QKV Projection + RoPE + KV Cache 写入。

收益：

- 减少 Kernel Launch。
- 减少中间张量写回和读取。
- 提高局部性。

风险：

- Kernel 过大造成寄存器压力、占用率下降。
- 动态 Shape 和分支使实现复杂。
- 融合后并行策略可能无法同时适合所有子算子。
- 数值计算顺序变化。

## 1.12 调度优化

调度决定“谁在何时、在哪个计算单元上处理哪块数据”。包括：

- Block / Core 划分。
- Warp / 线程布局。
- 循环顺序和展开。
- 数据布局。
- 流水 Stage 数。
- 多流、多请求和动态批处理。
- 通信与计算重叠。

调度优化本质是平衡并行度、数据复用、资源占用和同步成本。

## 1.13 Tiling

Tiling 是把大张量切成能放入片上高速存储的小块。以矩阵乘为例，将 M、N、K 维切成 `BM × BN × BK`：

- A Tile：`BM × BK`
- B Tile：`BK × BN`
- C Tile：`BM × BN`

Tile 太小：复用不足、Launch/循环开销高。  
Tile 太大：片上存储、寄存器不足，降低并发度或溢出。  
选择 Tile 时要考虑 Shape、数据类型、片上容量、矩阵指令粒度、核数和尾块比例。

## 1.14 端到端训练推理系统

端到端系统不只包括 Kernel，还包括：

- 数据加载和预处理。
- 框架图构建与编译。
- 运行时调度。
- 显存分配。
- 并行策略。
- 通信库。
- Checkpoint。
- 推理请求队列、动态批处理和 KV Cache 管理。
- 监控、容错和性能分析。

面试中应体现“局部优化必须服务于端到端指标”。

## 1.15 高性能算子库与基础加速库

- **高性能算子库**：面向 AI 工作负载，提供 GEMM、Conv、Attention、Norm、激活、Embedding 等实现。
- **基础加速库**：更底层，可能包含 BLAS、FFT、通信、内存管理、线程池、向量数学函数等。

它们通常需要多 Shape、多数据类型、多硬件版本的 Kernel 集合，以及自动选择、Autotune、Fallback 和回归测试。

## 1.16 Profiling

Profiling 是用数据定位瓶颈，不是凭感觉优化。至少分三层：

- **端到端层**：吞吐、TTFT、TPOT、显存、请求排队。
- **图/算子层**：热点算子、Launch 次数、通信比例、同步空洞。
- **Kernel 层**：计算单元利用率、带宽、Cache 命中、指令吞吐、占用率、分支和访存效率。

正确流程是：基线 → 定位 → 假设 → 修改 → Microbenchmark → 端到端验证 → 回归。

# 3. 大模型训练、微调与评估

## 3.1 Qwen 微调常见问题

可能遇到的问题及回答方向：

- **OOM**：减小 Micro Batch，使用梯度累积、Checkpoint、LoRA/QLoRA、ZeRO/FSDP、序列打包。
- **Loss 不降**：检查标签移位、Padding Mask、学习率、数据格式、Tokenizer、冻结参数和梯度。
- **Loss 震荡/NaN**：降低学习率，启用梯度裁剪，检查 FP16 溢出，改 BF16/FP32 累加，排查脏数据。
- **过拟合**：增加数据、去重、降低 Epoch、Weight Decay、早停。
- **能力遗忘**：混入通用数据、降低学习率、减少可训练参数、做多任务评估。
- **吞吐低**：提高序列打包率，优化 DataLoader，使用 FlashAttention、混合精度和分布式策略。

标准答法：

> 我会先区分数据、优化器、数值和系统四类问题。训练异常时先固定随机种子和小数据过拟合，确认计算图和标签正确；再检查梯度范数、学习率和混合精度；最后用 Profiler 看数据等待、通信和算子热点。这样避免一上来盲目调参。

## 3.2 训练参数如何设置？

- **Learning Rate**：最重要。全量预训练通常与 Batch、模型规模有关；微调一般更小。LoRA 常可略高于全参微调。
- **Warmup Ratio/Steps**：训练初期逐渐提高学习率，防止大梯度破坏预训练权重。
- **Batch Size**：影响吞吐、梯度噪声和泛化。全局 Batch = Micro Batch × 梯度累积 × 数据并行卡数。
- **Epoch**：数据少时容易过拟合，应由验证集决定。
- **Weight Decay**：抑制参数过大；Bias 和 Norm 参数常不做衰减。
- **Gradient Clipping**：限制梯度范数，防止爆炸。
- **Sequence Length**：直接影响 Attention 计算量和激活显存。
- **LoRA Rank `r`**：越大容量越高、显存和计算越多。
- **LoRA Alpha**：缩放 LoRA 更新，常见有效缩放为 `alpha / r`。
- **Dropout**：正则化；数据充足时可以较低。

回答重点不是报一个固定数，而是说明：

> 先从模型官方基线开始，在固定数据和评估集上做小规模搜索；一次只改变少量变量；记录 Loss、梯度范数、吞吐、显存和下游指标；最终根据验证集而不是训练 Loss 选参数。

## 3.3 如何评估微调模型？

四层评估：

1. **训练健康度**：Train/Val Loss、梯度范数、是否 NaN。
2. **任务指标**：Accuracy、F1、BLEU/ROUGE、Pass@k、检索 Recall 等。
3. **生成质量**：事实性、相关性、完整性、格式遵循、安全性；可用人工盲评或模型裁判，但要校准偏差。
4. **系统指标**：TTFT、TPOT、吞吐、显存、最大上下文长度和稳定性。

还要做基线对比、消融实验、分桶评估和回归集测试。

## 3.4 QLoRA 原理

QLoRA 的核心是：

- 将预训练基座权重量化到低比特，典型为 4 bit。
- 量化后的基座权重冻结。
- 在线计算时反量化参与前向。
- 只训练低秩适配矩阵 `ΔW = BA`。
- 通过低秩参数和低比特基座显著节省显存。

线性层可写为：

```text
y = (W_quant_dequant + s · B A) x
```

优点：显存低、训练参数少、接近全参微调效果。  
代价：反量化有开销；效果依赖 Rank、数据和目标模块；不等于真正的低比特训练。

## 3.5 LoRA 为什么有效？

经验假设是：下游任务所需的权重更新位于低维子空间，因此无需更新完整 `d_out × d_in` 矩阵，只训练两个小矩阵：

```text
ΔW = B A
A ∈ R^(r × d_in)
B ∈ R^(d_out × r)
r << min(d_in, d_out)
```

参数量从 `d_out × d_in` 降为 `r(d_in+d_out)`。

## 3.6 模型参数量和显存如何估算？

设参数量为 `P`：

- FP32 权重：约 `4P` 字节。
- FP16/BF16 权重：约 `2P` 字节。
- INT8 权重：约 `P` 字节，另加 Scale 等元数据。
- INT4 权重：约 `0.5P` 字节，另加元数据。

推理显存粗略为：

```text
权重 + KV Cache + 临时工作区 + 激活 + 框架/通信开销
```

Adam 混合精度训练的每参数显存通常远高于权重本身，因为还可能包含：

- FP16/BF16 参数。
- FP16/BF16 梯度。
- FP32 Master Weight。
- FP32 一阶矩。
- FP32 二阶矩。

仅这些状态粗略可达每参数 16 字节，实际还要加激活、临时张量、碎片和通信 Buffer。不要把“模型参数 × 2 字节”误当作训练总显存。

# 4. 大模型推理：Prefill、Decode、KV Cache

## 4.1 Prefill 与 Decode 的区别

**Prefill**：

- 一次处理整个 Prompt。
- Q、K、V 都来自多个 Token。
- GEMM 规模较大，并行度高。
- 通常更偏计算密集。
- 产生每层历史 K/V，写入 KV Cache。

**Decode**：

- 每轮通常只生成一个新 Token。
- Query 很短，但要读取全部历史 K/V。
- GEMM 更接近 GEMV 或小矩阵。
- 容易受显存带宽、Kernel Launch、调度和通信影响。
- 需要持续追加 KV Cache。

## 4.2 KV Cache 是什么？

自回归生成时，第 `t` 步只需计算新 Token 的 K/V，历史 Token 的 K/V 可复用。把历史 K/V 保存下来就是 KV Cache。

若每层有：

- Batch：`B`
- 历史长度：`S`
- KV Head 数：`H_kv`
- Head Dimension：`D`
- 层数：`L`
- 每元素字节：`bytes`

则 K 与 V 总大小近似：

```text
KV bytes = 2 × L × B × S × H_kv × D × bytes
```

如果使用 MHA，`H_kv = H_q`；GQA/MQA 通过减少 KV Head 数降低 Cache。

## 4.3 KV Cache 布局如何影响性能？

需要考虑：

- Decode 访问维度是否连续。
- 多请求分页后是否产生随机访问。
- Head、Token、Block 的排列顺序。
- 向量化和对齐。
- Cache Block 大小及碎片率。
- 张量并行下 KV 如何切分。

好的布局应使同一计算单元读取的数据连续、对齐，并减少地址计算和无效搬运。

## 4.4 Paged Attention

思想类似操作系统分页：KV Cache 不必为每个请求预留一块最大连续空间，而是按固定 Block 分配，通过 Block Table 找到物理位置。

优点：

- 减少预留和外部碎片。
- 支持请求动态增长。
- 更容易共享前缀和调度不同长度请求。

代价：

- 地址间接访问更复杂。
- Kernel 需要高效遍历 Block Table。

## 4.5 TTFT、TPOT、吞吐

- **TTFT**：Time To First Token，从请求到首 Token，主要受排队、Prefill 和调度影响。
- **TPOT/ITL**：生成阶段每个 Token 的平均时间，主要受 Decode 影响。
- **吞吐**：单位时间处理的 Token 或请求数。

提高批量通常提高吞吐，但可能恶化单请求时延，因此服务系统要在 SLA 下优化，而不是只追求 tokens/s。

## 4.6 Temperature、Top-k、Top-p

Temperature 对 Logits 缩放：

```text
p_i = exp(z_i / T) / Σ_j exp(z_j / T)
```

- `T < 1`：分布更尖锐，更确定。
- `T > 1`：分布更平坦，更随机。
- `T → 0`：接近 Argmax。

Top-k：仅保留概率最高的 k 个 Token。  
Top-p：按概率降序，保留累计概率达到 p 的最小集合。

常见执行逻辑是 Temperature 缩放 → 得到概率/排序依据 → 应用 Top-k、Top-p 过滤 → 重新归一化 → 采样。工程实现可能为了效率合并或调整顺序，面试时应说明“语义上先缩放，再过滤并归一化”。

# 5. 分布式训练：ZeRO、多卡切分与通信算子

## 5.1 ZeRO-1、2、3

- **ZeRO-1**：切分优化器状态。
- **ZeRO-2**：在 Stage 1 基础上再切分梯度。
- **ZeRO-3**：再切分模型参数，前向/反向时按需 AllGather，用完重新分片。

随着 Stage 提高，单卡显存更低，但通信、调度和实现复杂度增加。ZeRO-3 还可把状态 Offload 到 CPU/NVMe，但会受带宽和延迟制约。

## 5.2 数据并行

每张卡保存完整模型，读取不同数据。每卡计算本地梯度，再通过 AllReduce 得到全局梯度。

优点：简单、计算效率高。  
限制：单卡必须放得下完整模型和训练状态。

## 5.3 张量并行

将单层大矩阵按行或列切到多卡。例如 `Y = XW`：

- **列并行**：按 W 的输出列切分，每卡得到一部分输出。
- **行并行**：按 W 的输入行切分，每卡计算部分和，最后 Reduce。

张量并行通信频繁，适合节点内高带宽互联。

## 5.4 流水并行

按层切分模型，每个 Stage 放一段网络，Micro Batch 在 Stage 间流动。

问题：

- Pipeline Bubble。
- Stage 负载不均。
- 激活通信。
- 调度复杂。

通过增加 Micro Batch、交错调度和均衡切层降低气泡。

## 5.5 序列并行与上下文并行

- **序列并行**：把序列维上的计算或激活分散到多卡，常与张量并行结合以降低激活显存。
- **上下文并行**：把长上下文沿 Token 维切分，Attention 需要交换 K/V 或采用 Ring 等算法。

## 5.6 专家并行

MoE 将不同 Expert 放到不同卡，Token 根据 Router 发往对应 Expert。核心通信是 All-to-All。

瓶颈：

- Expert 负载不均。
- Token Dispatch/Combine 通信。
- 小 Expert GEMM 效率低。
- 容量限制导致 Token 丢弃或溢出。

优化包括负载均衡 Loss、Expert Placement、Token 重排、Grouped GEMM、通信计算重叠。

## 5.7 集合通信算子

- **Broadcast**：一个 Rank 向所有 Rank 发送。
- **Reduce**：多个 Rank 按 Sum/Max 等聚合到一个 Rank。
- **AllReduce**：所有 Rank 获得聚合结果，数据并行梯度同步常用。
- **AllGather**：每卡持有一片，最终所有卡拿到全部片段。
- **ReduceScatter**：先归约，再把结果分片给各卡；可看作 AllReduce 的前半部分。
- **All-to-All**：每个 Rank 向所有 Rank 发送不同数据，MoE 常用。

## 5.8 Ring AllReduce

典型实现分两步：

1. ReduceScatter：每卡经过多轮发送和归约，最终保留结果的一片。
2. AllGather：再经过多轮交换，让所有卡获得完整结果。

优点是带宽利用率高、每卡传输量相对均衡；小消息时延和拓扑不匹配时未必最优。

## 5.9 如何重叠通信与计算？

- 梯度按 Bucket 分组，某 Bucket 反向完成后立即发起通信。
- 使用异步 Collective 和独立 Stream。
- 将大通信切成块，流水执行。
- 合理调整 Bucket 大小：太小启动开销高，太大重叠不足。
- 避免主线程或默认流上的隐式同步。

## 5.10 “矩阵 A×B 用 4 张卡如何切？”

设 `A[M,K] × B[K,N]`：

- 按 N 切 B：每卡持有 `B[:, N/4]`，计算 `C` 的列块；输入 A 可能复制或 AllGather。
- 按 K 切：每卡持有 A 和 B 的 K 分片，产生部分和，最后 AllReduce/ReduceScatter。
- 2D 切分：4 卡组成 `2×2` 网格，同时切 M、N 或 K，减少单卡存储与通信峰值，但实现更复杂。

选择取决于张量初始分布、后续算子是否能消费分片结果、通信拓扑和矩阵形状。高分回答一定提到“避免不必要地 Gather 完整结果，让分片布局跨层传播”。

# 6. Transformer、Attention 与大模型核心算子

## 6.1 Transformer 架构

典型 Decoder-only 大模型每层包含：

1. Norm。
2. Q/K/V 线性投影。
3. 位置编码（如 RoPE）。
4. Causal Self-Attention。
5. 输出投影与残差。
6. Norm。
7. MLP（如 SwiGLU）。
8. 残差。

训练使用 Teacher Forcing 和 Causal Mask；推理按 Token 自回归生成并使用 KV Cache。

## 6.2 多头注意力公式

```text
Q = XW_Q, K = XW_K, V = XW_V
Attention(Q,K,V) = softmax(QK^T / sqrt(d_k) + mask) V
```

多头将隐藏维拆成多个 Head，各 Head 独立做 Attention，拼接后再线性映射。

为什么除以 `sqrt(d_k)`：如果 Q、K 元素方差近似为 1，点积方差随 `d_k` 墑大；缩放可避免 Softmax 输入过大而饱和、梯度过小。

## 6.3 位置编码

纯 Attention 对输入排列本身没有顺序感知，需要显式注入位置。

- **绝对位置 Embedding**：位置 ID 映射为向量。
- **相对位置 Bias**：根据 Token 相对距离加入 Attention Logit。
- **RoPE**：对 Q/K 的二维通道对施加随位置变化的旋转，使点积自然包含相对位置信息。

## 6.4 RoPE

把每对通道视为二维向量，根据位置 `m` 和频率 `θ_i` 旋转：

```text
[x'_(2i)  ]   [ cos(mθ_i) -sin(mθ_i)] [x_(2i)  ]
[x'_(2i+1)] = [ sin(mθ_i)  cos(mθ_i)] [x_(2i+1)]
```

实现关注：

- Sin/Cos 表缓存。
- Q/K Layout。
- 向量化。
- 与 QKV 投影、KV Cache 写入融合。

## 6.5 MHA、MQA、GQA

- **MHA**：Q、K、V Head 数相同。
- **MQA**：多个 Query Head 共享一组 K/V Head，KV Cache 最小。
- **GQA**：若干 Query Head 共享一个 KV Head，在质量和 Cache 之间折中。

## 6.6 Softmax 数值稳定实现

直接 `exp(x)` 可能溢出。使用：

```text
m = max(x)
softmax_i = exp(x_i - m) / Σ_j exp(x_j - m)
```

并行实现需要 Max Reduction、Exp、Sum Reduction、Normalize。长向量时要分块并维护全局统计量。

## 6.7 在线 Softmax

处理新分块时维护当前最大值 `m` 和指数和 `l`。若旧块统计为 `(m_old, l_old)`，新块最大值为 `m_new_block`：

```text
m_new = max(m_old, m_new_block)
l_new = exp(m_old - m_new) * l_old
      + Σ exp(x_new - m_new)
```

这样无需保存完整 Logits，可用于 FlashAttention 的分块计算。

## 6.8 FlashAttention 为什么快？

普通 Attention 往往显式生成 `S×S` 的注意力矩阵并反复读写 HBM。FlashAttention：

- 将 Q、K、V 分块搬入片上存储。
- 在线维护 Softmax 统计量。
- 直接累积输出。
- 不物化完整注意力矩阵。

它不是降低主要 FLOPs，而是降低外部内存 IO，因此在 Attention 受带宽限制时显著加速。

## 6.9 LayerNorm 与 RMSNorm

LayerNorm：

```text
μ = mean(x)
σ² = mean((x-μ)²)
y = γ (x-μ) / sqrt(σ²+ε) + β
```

RMSNorm：

```text
rms = sqrt(mean(x²)+ε)
y = γ x / rms
```

RMSNorm 不减均值，计算更少，在许多大模型中使用。两者都是归约 + 逐元素计算，优化重点是减少遍历和融合。

## 6.10 GELU 与 SwiGLU

GELU 是平滑激活：

```text
GELU(x) = x Φ(x)
```

SwiGLU 常写为：

```text
SwiGLU(x) = SiLU(xW_gate) ⊙ (xW_up)
```

通常将两个投影合成一个更大的 GEMM，随后融合切分、SiLU 和逐元素乘。

## 6.11 GEMM 性能关键点

对于 `C[M,N] = A[M,K] × B[K,N]`：

- Tile M/N/K。
- 数据布局与连续访存。
- 片上复用 A/B Tile。
- 使用矩阵乘专用指令。
- 累加精度。
- 双缓冲和流水。
- 尾块处理。
- Batch 和小矩阵调度。
- Epilogue 融合 Bias/Activation/Quant。

## 6.12 MoE

MoE 每个 Token 只激活少数 Expert：

1. Router 计算 Expert 得分。
2. Top-k 选择 Expert。
3. Token 按 Expert 重排和跨卡分发。
4. Expert MLP 计算。
5. 合并并按路由权重加权。

热点算子：TopK、Histogram/Prefix Sum、Scatter/Gather、All-to-All、Grouped GEMM。

# 7. Ascend C、Triton、TileLang

## 7.1 昇腾（Ascend）

昇腾是面向 AI 训练和推理的计算平台。理解“昇腾”时不要只想到一张加速卡，它包含多个层次：

- **AI 处理器 / NPU**：执行矩阵、向量、标量和数据搬运。
- **Atlas 硬件产品**：训练服务器、推理服务器、加速卡、边缘设备等。
- **CANN 软件栈**：编译器、运行时、算子库、图引擎、通信与开发工具。
- **算子开发接口**：例如 Ascend C。
- **框架适配**：MindSpore、PyTorch 等框架通过插件或后端使用昇腾能力。

面试时可回答：

> 昇腾是从芯片到系统软件再到框架适配的完整 AI 计算平台。算子岗位于其中的关键路径：把框架里的算子语义映射为适合 AI Core 存储层次和计算单元的数据流与并行实现。

## 7.2 NPU

NPU 是 Neural Processing Unit，针对神经网络中高密度矩阵、向量和张量计算设计的处理器。

与通用 CPU 相比，NPU：

- 拥有更高的张量计算吞吐。
- 更强调片上存储和显式数据搬运。
- 对规则、大规模并行运算效率高。
- 对复杂控制流和不规则访存未必占优。

与 GPU 相比，NPU 同样面向并行计算，但会针对 AI 运算和数据类型做更强的专用化。高性能算子必须理解硬件支持的矩阵指令粒度、向量单元、片上 Buffer、核间并行和数据搬运路径。

## 7.3 CANN

CANN 是昇腾 AI 异构计算软件栈。可将它理解为连接上层 AI 框架与底层昇腾硬件的系统软件集合，通常涉及：

- 图编译和优化。
- 算子编译与执行。
- 运行时与任务调度。
- 内存管理。
- 高性能算子库。
- 集合通信。
- Profiling、调试和部署工具。

算子开发者写出的 Kernel 不是孤立运行的，它需要经过 CANN 的编译、加载、调度和运行时管理。

## 7.4 Ascend C

Ascend C 是 CANN 面向算子开发推出的编程语言，原生支持 C/C++ 标准规范。程序经过编译器编译和运行时调度后运行在昇腾 AI 处理器上。

核心理解：

- 使用接近 C++ 的方式表达数据搬运和张量计算。
- 开发者负责把全局张量切成 Tile。
- 将数据从外部存储搬到片上存储。
- 在 AI Core 上执行向量、矩阵或归约计算。
- 通过多核切分和流水提高性能。

一个典型 Ascend C Kernel 的逻辑可以抽象为：

```text
Init:
  计算当前核负责的数据范围
  初始化全局张量和片上队列

Process:
  for each tile:
    CopyIn   外部存储 -> 片上存储
    Compute  片上计算
    CopyOut  片上存储 -> 外部存储
```

### Ascend C 中的 Host 与 Kernel

- **Host 侧**：做 Shape 推导、Tiling 计算、参数准备、Kernel Launch。
- **Kernel 侧**：运行在 AI Core 上，执行数据搬运和计算。

Host 决定“如何切”和“启动多少核”；Kernel 决定“每个核如何处理自己的 Tile”。

### Global Memory / Local Memory

- **Global Memory**：容量大、访问延迟高，是输入输出张量所在的外部存储。
- **Local / On-chip Memory**：容量小、速度快，供计算单元使用。

性能优化的核心通常是：让数据尽量少往返 Global Memory，并在片上被重复使用。

### Queue 与双缓冲

Queue 用于组织不同流水阶段之间的数据。使用多个 Buffer 后，可以在计算当前 Tile 时搬运下一个 Tile，实现 Copy 与 Compute 重叠。

### 多核切分

一般根据总元素数或高维块，把工作分给多个 AI Core：

```text
block_length = ceil_div(total_work, block_num)
start = block_idx * block_length
```

还必须处理尾核工作不足、非整除、核间负载不均和对齐。

## 7.5 Triton

Triton 是面向并行计算 Kernel 的领域专用语言和编译器。它以 Python 风格写 Kernel，但执行的不是普通 Python；代码会被编译为设备程序。

Triton 的核心思想是 **Blocked Program**：

- CUDA 常从“一个线程处理一个或少量标量”出发。
- Triton 更常从“一个 Program Instance 处理一块数据”出发。
- 编译器再把块级表达映射到线程、向量指令、共享存储和矩阵单元。

### Program

Program 是一次块级 Kernel 实例。大量 Program 并行覆盖完整数据。

### `program_id`

`program_id(axis)` 类似块编号，用来确定当前 Program 处理哪个 Tile：

```text
pid = program_id(0)
offsets = pid * BLOCK_SIZE + arange(0, BLOCK_SIZE)
```

### Block

Block 是 Program 一次处理的数据块。Block Size 会影响：

- 并行度。
- 数据复用。
- 寄存器和共享存储占用。
- 尾块浪费。

### Mask

当数据长度不能整除 Block Size 时，使用 Mask 防止越界读写：

```text
mask = offsets < n
x = load(ptr + offsets, mask=mask)
store(out + offsets, y, mask=mask)
```

### Triton 编译器会做什么？

根据官方编程模型，编译器可参与完成合并访存、向量化、预取、共享存储分配与同步、异步拷贝调度、适配矩阵计算单元的指令选择等。

但编译器不能替开发者做完全部性能决策。开发者仍要选择 Tile、Program 映射、流水 Stage、Warp 数量和数据布局。

### Triton 与 CUDA 的区别

| 维度 | CUDA | Triton |
|---|---|---|
| 抽象层次 | 线程、Warp、Block | Program、张量 Block |
| 控制能力 | 最细粒度、最灵活 | 更高层、更容易开发 |
| 开发速度 | 相对慢 | 相对快 |
| 极限优化 | 可手工控制每个细节 | 依赖编译器与 DSL 表达能力 |
| 适用场景 | 任意 GPU 程序 | AI 中规则的张量 Kernel 尤其合适 |

## 7.6 TileLang

TileLang 是面向高性能张量程序的 Tile 级编程语言和编译系统。其文档包含语言基础、软件流水标注、自动调优、GEMM/GEMV、稀疏矩阵乘和 FlashMLA 等内容。

核心思想：

- 以 Tile 为中心表达矩阵和张量计算。
- 显式描述片上 Buffer、拷贝、计算和流水。
- 通过高层原语表达 GEMM、归约等操作。
- 让编译器降低到目标后端，并提供自动调优能力。

TileLang 适合“比算子库更灵活、比手写底层 Kernel 更高效开发”的位置。

## 7.7 Tile 抽象

Tile 是张量的一个规则块。Tile 抽象把开发者关心的对象从单个标量或线程，提升为：

- 一个矩阵子块。
- 一个向量段。
- 一个张量局部区域。

这样更容易直接表达：

- 片上复用。
- 块矩阵乘。
- 流水拷贝。
- 布局转换。
- 分块归约。

## 7.8 Ascend C、Triton、TileLang 如何比较？

| 维度 | Ascend C | Triton | TileLang |
|---|---|---|---|
| 主要目标 | 昇腾算子开发 | GPU 张量 Kernel | Tile 级高性能张量程序 |
| 风格 | C/C++ | Python DSL | Python/TVM 风格 DSL |
| 核心抽象 | 数据搬运、队列、片上计算 | Blocked Program | Tile、Buffer、Pipeline |
| 硬件控制 | 面向昇腾，较直接 | 编译器承担较多映射 | 显式 Tile 和流水，兼顾抽象与控制 |
| 岗位意义 | 直接开发昇腾算子 | 理解现代 DSL 与 GPU Kernel | 理解新型 Tile DSL、自动调优和算子生成 |

高分回答：

> 三者共同目标都是用比通用框架算子更低的层次表达高性能 Kernel，但抽象不同。Ascend C 更贴近昇腾的数据搬运和 AI Core 执行模型；Triton 用 Blocked Program 提升 GPU Kernel 开发效率；TileLang 用 Tile、Buffer 和软件流水表达张量程序。真正需要掌握的是它们背后的共同原则：分块、布局、片上复用、并行映射、流水和精度。

## 7.9 DSL

DSL 是 Domain-Specific Language，领域专用语言。它不追求解决所有编程问题，而是对某一领域提供更合适的抽象。

AI Kernel DSL 的价值：

- 比通用语言更容易识别张量语义。
- 编译器能进行领域特定优化。
- 降低手写底层 Kernel 的复杂度。
- 便于自动生成和自动调优。

代价是表达范围、生态、调试工具和跨硬件可移植性可能受限。

# 8. 算子开发、调测与性能优化

## 8.1 算子设计

算子设计阶段先确定“算什么”，还不是立刻写 Kernel：

1. 数学定义。
2. 输入输出 Shape 和数据类型。
3. 广播、Padding、Mask 规则。
4. 动态 Shape 范围。
5. 空张量、NaN、Inf 等边界语义。
6. 反向传播公式。
7. 精度容差。
8. 目标硬件和性能指标。

如果语义不清晰，后续优化越深，错误越难发现。

## 8.2 算子编码

编码通常分为：

- **参考实现**：可读、正确，常用 Python/NumPy/PyTorch 编写。
- **Host 代码**：参数校验、Shape 推导、Tiling、Launch。
- **Kernel 代码**：多核切分、CopyIn、Compute、CopyOut。
- **框架注册**：让上层图和运行时能发现、选择并执行该算子。

## 8.3 算子测试

测试至少包括：

- **功能测试**：与参考实现比较。
- **Shape 测试**：小 Shape、大 Shape、非整除、动态 Shape、空张量。
- **数据类型测试**：FP32、FP16、BF16、INT8 等。
- **数据分布测试**：随机、全零、极值、长尾、重复值。
- **异常测试**：非法参数、越界、NaN/Inf。
- **精度测试**：绝对误差、相对误差、ULP。
- **性能测试**：时延、吞吐、带宽、利用率。
- **稳定性测试**：重复执行、并发、长时间运行。
- **回归测试**：防止新优化破坏旧 Shape 或精度。

## 8.4 调测

“调测”是调试与测试的合称，包含：

- 编译错误排查。
- Kernel 运行错误和越界定位。
- 中间结果比对。
- 多核切分检查。
- 同步、竞争和死锁排查。
- 数值误差定位。
- 性能数据采集。

正确做法是先用小 Shape 和确定性数据验证，再扩大规模和并发。

## 8.5 性能基线

没有基线就无法证明优化有效。基线可选择：

- 框架原生实现。
- 现有算子库。
- 朴素 Kernel。
- 理论 Roofline 上限。

记录时应固定：

- 硬件和软件版本。
- Shape、数据类型、布局。
- 预热次数。
- 重复次数和统计方法。
- 功耗、频率和并发环境。

## 8.6 Microbenchmark

Microbenchmark 是只测一个 Kernel 或很小子图的基准。用途是隔离框架、数据加载和业务调度的影响。

注意：

- 先预热。
- 设备执行通常异步，计时前后要正确同步。
- 不要把编译时间算进稳态执行时间，除非研究首运行时延。
- 报告中位数、P90/P99 或稳定区间，不只报最好一次。
- 仍要回到端到端验证，避免“微基准变快，系统反而变慢”。

## 8.7 Roofline 模型

Roofline 用算术强度判断理论瓶颈：

```text
Arithmetic Intensity = FLOPs / Bytes moved
Attainable Performance
  <= min(Peak Compute, Memory Bandwidth × Arithmetic Intensity)
```

- 位于斜线区域：带宽受限。
- 位于水平区域：计算受限。

优化方向：

- 带宽受限：减少访存、融合、提高数据复用。
- 计算受限：使用更高效指令、提高矩阵单元利用率、减少无效计算。

## 8.8 FLOPs

FLOPs 在不同语境有两种含义：

- **Floating Point Operations**：一次任务的浮点运算数量。
- **FLOPS**：每秒浮点运算能力。

矩阵乘 `M×K` 乘 `K×N` 的运算量常近似为：

```text
2 × M × N × K
```

一次乘加按 2 个浮点操作计。

## 8.9 带宽

带宽是单位时间可传输的数据量。算子有效带宽可估算：

```text
effective_bandwidth = required_bytes / elapsed_time
```

需要区分理论带宽和有效带宽。访问不连续、缓存未命中、对齐差、请求粒度小都会降低有效带宽。

## 8.10 时延、吞吐和利用率

- **时延**：一次操作完成所需时间。
- **吞吐**：单位时间完成的操作、样本或 Token 数。
- **利用率**：硬件某资源忙碌程度。

利用率高不必然性能好。例如大量无效计算也能让计算单元繁忙；必须结合实际吞吐、带宽和理论工作量。

## 8.11 Latency-bound、Bandwidth-bound、Compute-bound

- **Latency-bound**：工作太小、依赖链太长或 Launch 过多，硬件无法充分并行。
- **Bandwidth-bound**：主要时间花在数据搬运。
- **Compute-bound**：主要时间花在计算指令。

小 Batch Decode 常同时有带宽和 Launch/调度问题；大 GEMM 更可能计算受限。

## 8.12 Kernel Launch

Host 向设备提交一个 Kernel 有固定开销。算子太碎时，即使每个 Kernel 很快，总 Launch 开销也会显著。

优化：

- 算子融合。
- CUDA Graph/静态执行图一类机制。
- 持久化 Kernel。
- 合并小任务和批处理。

## 8.13 合并访存（Coalesced Access）

相邻线程或并行 Lane 访问连续、对齐的地址，使硬件用更少内存事务完成读取。

反例：跨大步长访问、AoS 布局中只读取一个字段、转置访问。  
优化：改变 Layout、向量化 Load/Store、Tile 转置、SoA 布局。

## 8.14 对齐（Alignment）

数据地址和访问长度满足硬件要求时，可使用更宽的内存事务和向量指令。

对齐策略：

- 首地址对齐。
- 行跨度对齐。
- Tile 大小对齐指令粒度。
- 对尾部使用 Mask 或标量路径。

## 8.15 Bank Conflict

片上共享存储常划分为多个 Bank。若同一周期多个线程访问同一 Bank 的不同地址，请求可能串行化。

优化：

- Padding。
- Swizzle。
- 调整 Tile Layout。
- 改变访问顺序。

## 8.16 寄存器压力

每个线程或执行单元使用过多寄存器，会降低可同时驻留的 Warp/Program 数；严重时寄存器溢出到更慢的存储。

来源：

- Tile 太大。
- 循环展开过度。
- 中间值生命周期太长。
- 融合过深。

## 8.17 Occupancy

Occupancy 表示硬件上实际驻留的执行单元数量相对于上限的比例，受寄存器、共享存储、Block 大小等限制。

Occupancy 不是越高越好。若数据复用非常好，较低 Occupancy 也可能达到更高性能；它只是判断延迟隐藏能力的一个指标。

## 8.18 尾块（Tail Tile）

Shape 不能整除 Tile 时出现尾块。处理方式：

- Mask。
- Padding。
- 专门的 Tail Kernel。
- 根据 Shape 选择不同 Tile。

尾块比例大时，大 Tile 可能导致大量无效计算。

## 8.19 动态 Shape

运行时才能确定的 Shape。难点：

- 编译期无法完全特化。
- 最优 Tile 随 Shape 变化。
- 内存规划更复杂。
- 控制流和边界更多。

实践中常做 Shape 分桶、Kernel 多版本、运行时 Dispatch 或 JIT 编译。

## 8.20 Autotuning

自动调优是在候选参数空间中搜索最佳配置，例如：

- Tile M/N/K。
- 核数或 Program 数。
- Warp 数。
- 流水 Stage。
- 数据布局。
- 向量宽度。

搜索方式可为网格、随机、贝叶斯优化或代价模型。必须有编译缓存、正确性检查和稳定基准，否则搜索结果不可信。

## 8.21 性能调测方法

推荐面试回答：

> 我会先建立可复现基线，用端到端 Trace 找出热点，再对热点做 Microbenchmark。之后用 Roofline 判断计算还是带宽受限，并检查 Launch、并行度、访存合并、片上复用、寄存器和负载均衡。每次只验证一个主要假设，完成精度与性能回归，最后再确认端到端指标。

# 9. 精度分析、混合精度与量化

## 9.1 浮点数

浮点数通常由符号位、指数和尾数组成。有限位数导致：

- 很多实数无法精确表示。
- 加法和乘法存在舍入。
- 不满足严格结合律：`(a+b)+c` 可能不等于 `a+(b+c)`。
- 存在溢出、下溢、NaN、Inf 和非规格化数。

因此并行归约、融合和等价变换可能改变结果的最低若干位。

## 9.2 FP32、FP16、BF16

| 类型 | 总位数 | 指数范围 | 尾数精度 | 特点 |
|---|---:|---:|---:|---|
| FP32 | 32 | 大 | 高 | 精度好、吞吐和带宽成本较高 |
| FP16 | 16 | 较小 | 较高于 BF16 | 范围小，训练时更易溢出 |
| BF16 | 16 | 接近 FP32 | 较低 | 动态范围大，适合训练 |

FP16 与 BF16 都可减少显存和带宽并使用低精度计算单元。BF16 保留与 FP32 接近的指数位，更不容易溢出；FP16 尾数更多，局部表示精度更高。

## 9.3 混合精度

混合精度不是把所有张量都改成 FP16，而是：

- 大型矩阵乘使用 FP16/BF16 输入。
- 累加常使用更高精度。
- 敏感算子保留 FP32。
- 优化器状态或 Master Weight 可保留 FP32。

目标是在性能、显存和数值稳定性之间平衡。

## 9.4 Loss Scaling

FP16 能表示的最小有效梯度有限，小梯度可能下溢为 0。Loss Scaling 在反向前把 Loss 乘以比例 `S`，使梯度整体放大；更新前再除以 `S`。

- **静态 Scaling**：固定 S。
- **动态 Scaling**：若检测到 Inf/NaN 就降低 S，稳定一段时间后提高 S。

BF16 动态范围更大，通常对 Loss Scaling 依赖较小。

## 9.5 量化

量化把高精度数映射到低比特整数或浮点表示，减少模型大小、带宽和计算成本。

典型线性量化：

```text
q = clamp(round(x / scale) + zero_point)
x_hat = scale × (q - zero_point)
```

- `scale` 决定量化步长。
- `zero_point` 让实数 0 可精确映射到整数值。

## 9.6 对称量化与非对称量化

- **对称量化**：`zero_point = 0`，范围围绕 0 对称，计算简单。
- **非对称量化**：允许非零 Zero Point，更好利用偏移分布的整数范围，但计算和校准更复杂。

权重常用对称量化；激活可能使用非对称量化。

## 9.7 Per-tensor、Per-channel、Per-group

- **Per-tensor**：整个张量共用一个 Scale，元数据少但精度较差。
- **Per-channel**：每个输出通道一个 Scale，精度更好。
- **Per-group**：每若干元素一个 Scale，是低比特大模型权重量化常见折中。

Group 越小，适应局部分布能力越强，但 Scale 元数据和反量化开销越高。

## 9.8 PTQ 与 QAT

- **PTQ**：Post-Training Quantization，训练后量化。成本低，需要校准数据。
- **QAT**：Quantization-Aware Training，在训练中模拟量化误差，模型可适应低比特，通常精度更好。

## 9.9 权重量化与激活量化

- **Weight-only Quantization**：只量化权重；Decode 常受权重带宽限制，因此收益明显，实现相对容易。
- **Weight-Activation Quantization**：权重和激活都低比特，可获得更高计算吞吐，但校准与精度控制更难。
- **KV Cache Quantization**：降低长上下文推理的 Cache 容量和带宽，但要控制 Attention 误差。

## 9.10 量化 GEMM

量化 GEMM 可能包含：

1. 读取压缩权重。
2. 解包 INT4/INT8。
3. 读取 Scale/Zero Point。
4. 反量化或使用整数矩阵指令。
5. 高精度累加。
6. 应用 Bias、激活或再次量化。

关键是将解包、反量化和 GEMM 流水化或融合，避免低比特节省的带宽被额外 Kernel 开销抵消。

## 9.11 绝对误差与相对误差

```text
absolute_error = |actual - reference|
relative_error = absolute_error / max(|reference|, epsilon)
```

- 参考值接近 0 时，相对误差会失真，应同时看绝对误差。
- 大数值区间只看绝对误差也不公平，应结合相对误差。

## 9.12 ULP

ULP 是 Units in the Last Place，表示两个浮点数在可表示数轴上相差多少个最小单位。它能从浮点表示角度衡量误差，但不同数量级、特殊值和不同数据类型需要谨慎解释。

## 9.13 余弦相似度

用于衡量两个向量方向是否一致：

```text
cos(x,y) = (x·y) / (||x|| ||y||)
```

对大张量整体趋势很有用，但可能掩盖少数严重错误，因此不能单独作为正确性标准。

## 9.14 数值稳定

数值稳定是指算法在有限精度计算中不会因为微小舍入误差产生巨大偏差。

常见技术：

- Softmax 减最大值。
- LogSumExp。
- Welford 算法计算方差。
- FP32 累加。
- Kahan Summation 补偿求和。
- 避免两个接近的大数相减。
- 合理设置 Epsilon。

## 9.15 Welford 算法

Welford 在线更新均值和方差，数值稳定且适合分块、并行合并。相比 `E[x²]-E[x]²`，它不容易因两个接近大数相减产生严重消减误差。

## 9.16 精度对齐流程

1. 建立 FP32/FP64 参考实现。
2. 固定随机种子和输入。
3. 从小 Shape 开始逐步放大。
4. 保存关键中间张量。
5. 找到首个显著偏差点。
6. 检查数据类型转换、归约顺序和边界。
7. 对不同分布和极值做压力测试。
8. 评估算子误差对最终模型指标的影响。

# 10. AI Agent 与自动算子生成

## 10.1 AI Agent

AI Agent 是能围绕目标进行感知、推理、规划、调用工具、观察结果并迭代执行的系统。它不等于一次大模型对话。

一个典型 Agent 循环：

```text
用户目标
  -> 规划
  -> 选择工具
  -> 执行
  -> 观察结果
  -> 更新状态/记忆
  -> 继续或结束
```

## 10.2 Agent 架构

常见组件：

- **模型**：理解任务、推理和生成行动。
- **Planner**：拆分任务、排序依赖。
- **Tool Registry**：描述可调用的编译器、测试、Profiler、代码库等工具。
- **Executor**：执行工具调用。
- **Memory**：保存上下文、历史经验和失败案例。
- **Evaluator/Critic**：判断结果是否正确、是否达到目标。
- **Guardrail**：权限、资源、超时和安全控制。

## 10.3 ReAct

ReAct 将推理与行动交替：

```text
Reason -> Act -> Observe -> Reason -> ...
```

优势是模型可根据真实工具反馈修正计划，而不是一次生成完整答案。用于算子开发时，可让 Agent 生成代码、编译、运行测试、读取错误、修改代码和重新评测。

## 10.4 CoT

CoT 是 Chain of Thought，链式推理。它通过中间步骤帮助模型解决复杂问题。

工程上更应关注：

- 是否能分解问题。
- 是否能输出可验证的中间产物。
- 是否能调用工具验证。

不能把自然语言推理当作正确性证明；编译器、测试和性能数据才是算子开发的最终依据。

## 10.5 Agent 记忆

- **短期记忆**：当前任务上下文、代码和测试结果。
- **长期记忆**：历史成功 Kernel、硬件约束、常见错误。
- **情景记忆**：某个 Shape/算子优化过程。
- **语义记忆**：文档、API、最佳实践。

实现可使用结构化数据库、向量检索、代码索引和实验数据库。写入长期记忆前应去噪和验证，避免错误经验污染后续任务。

## 10.6 自动算子生成

自动算子生成是从高层算子描述出发，自动产生目标硬件 Kernel。输入可能包括：

- 数学表达式或计算图。
- Shape 和数据类型。
- 目标硬件约束。
- 精度与性能目标。

输出包括：

- Kernel 代码。
- Tiling 和调度参数。
- Host 侧配置。
- 测试和性能报告。

## 10.7 基于规则的生成

使用预先定义的模板和 Rewrite：

- 识别算子模式。
- 选择已有 Schedule。
- 替换参数和 Shape。
- 编译并运行。

优点是可控、可解释；缺点是覆盖有限，规则维护成本高。

## 10.8 搜索式生成

将 Schedule 参数作为搜索空间，通过自动调优选择最优候选：

- Tile。
- 并行映射。
- 循环顺序。
- Pipeline Stage。
- 数据布局。

评价函数通常是正确性约束下的实际运行时间，也可用代价模型降低测量次数。

## 10.9 LLM/Agent 驱动生成

Agent 可执行：

1. 读取算子规范和硬件文档。
2. 检索相似实现。
3. 生成 Kernel。
4. 编译和修复错误。
5. 生成测试输入并对齐参考实现。
6. 运行 Profiler。
7. 根据瓶颈修改 Tile、布局和流水。
8. 输出报告和提交候选代码。

## 10.10 自动生成的正确性保障

必须建立硬约束：

- 编译通过。
- 多 Shape、多类型单元测试。
- 与参考实现误差对齐。
- Sanitizer/越界检测。
- 随机测试和属性测试。
- 性能回归。
- 人工 Review 后才能进入生产。

“模型认为正确”不构成验证。

## 10.11 自动生成的性能评价

至少比较：

- 现有库基线。
- 朴素实现。
- 理论上限。
- 不同 Shape 分桶。
- 编译时间和运行时间。
- 端到端收益。

还要惩罚只对单一 Shape 过拟合、代码体积过大、编译耗时过长和稳定性差的方案。

## 10.12 算子开发自动化程度

自动化不一定意味着完全无人：

- Level 1：自动生成模板和测试。
- Level 2：自动修复编译错误。
- Level 3：自动搜索调度并生成报告。
- Level 4：自动完成正确性与性能闭环，人类 Review。
- Level 5：在严格权限和回归系统下自动合入或部署。

现实中更可行的是“Agent 提升工程师效率”，而不是取消工程师。

# 11. 岗位要求中的专业与学科

## 11.1 人工智能（Artificial Intelligence）

人工智能是让计算机完成通常需要人类智能的任务的总称，包括感知、学习、推理、规划、决策和生成。

- 数值优化。
- 数值积分和微分方程。
- 误差分析。
- 迭代方法。

## 11.5 应用数学

应用数学把数学模型用于实际问题，例如优化、概率、图论、信号处理和控制。它强调从业务或物理问题抽象出可计算模型。

岗位中常用：

- 优化方法。

- 编码规范。
- 测试。
- 版本管理。
- CI/CD。
- Code Review。
- 可维护性和可扩展性。

计算机科学涵盖算法、数据结构、操作系统、编译原理、体系结构、网络、数据库、人工智能等。

模型算法 -> AI 框架 -> 计算图/编译器 -> Kernel -> 运行时 -> 硬件

# 12. 大模型架构、AI 框架与 AI 系统

## 12.1 大模型

大模型通常指参数规模、训练数据和计算量显著增大的基础模型，能够通过预训练学习通用表示，再通过微调、提示或工具调用适配任务。

“大”带来的系统变化：

- 单卡放不下，需要分布式并行。
- 训练状态和激活占用巨大。
- 通信成为瓶颈。
- 推理需要 KV Cache、动态批处理和量化。
- 单个核心算子的优化会被放大到巨大算力成本。

## 12.2 大模型架构

架构是模型组件及其连接方式。当前语言大模型常采用 Decoder-only Transformer，包含：

- Tokenizer 与 Embedding。
- 多层 Transformer Block。
- Attention。
- MLP。
- Norm 和残差。
- 输出词表投影。

进一步变化包括 GQA/MQA、RoPE、RMSNorm、SwiGLU、MoE、稀疏 Attention、多模态编码器和长上下文机制。

## 12.3 AI 框架

AI 框架为模型开发提供张量、自动微分、算子、模块、优化器、分布式训练和部署接口，例如 PyTorch、TensorFlow、MindSpore。

主要层次：

- Python 前端 API。
- 张量与算子分发。
- 自动微分。
- 计算图捕获。
- 编译与图优化。
- 设备后端。
- Kernel 库。
- 分布式运行时。

算子开发需要把自定义 Kernel 注册到框架，使框架能做 Shape 推导、反向传播和设备调度。

## 12.4 PyTorch

PyTorch 以动态图和易用性著称，核心概念：

- Tensor。
- Autograd。
- `nn.Module`。
- Optimizer。
- Dispatcher。
- Device Backend。
- Torch Compile / 图捕获和编译。

了解源码时可重点看算子如何从 Python API 经 Dispatcher 到不同设备实现，以及 Autograd 如何注册反向。

## 12.5 TensorFlow

TensorFlow 强调数据流图、跨设备执行和部署生态。早期静态图需要先构图后执行，现代版本也支持 Eager 模式和图编译。

静态图优点是全局优化和部署方便；缺点是动态调试和复杂控制流表达相对困难。现代框架通常在动态图易用性和图编译性能之间结合。

## 12.6 MindSpore

MindSpore 是面向全场景 AI 的框架，提供张量、自动微分、模型构建、分布式并行和昇腾适配。对该岗位而言，重点不是背 API，而是理解框架如何把模型图下沉到 CANN 和高性能算子。

## 12.7 自动微分

自动微分不是数值差分，也不是符号推导。它把程序拆成基本运算，并应用链式法则精确计算导数。

- **前向模式**：沿计算方向传播导数，适合输入维度少。
- **反向模式**：从输出向输入传播梯度，适合标量 Loss 对大量参数求导，是深度学习主流。

算子需要定义正确的反向公式，或能被框架用已有算子组合求导。

## 12.8 计算图

计算图中：

- 节点表示算子。
- 边表示张量和依赖。

动态图运行时构建，灵活易调试；静态图提前构建，便于全局优化。现代编译通常尝试捕获动态图的稳定部分形成图。

## 12.9 编译器前端、中端和后端

- **前端**：解析框架程序，生成中间表示。
- **中端**：进行与硬件相对无关的图优化和算子融合。
- **后端**：做目标硬件相关的指令选择、内存规划、调度和代码生成。

## 12.10 IR

IR 是 Intermediate Representation，中间表示。它用结构化形式表达计算，便于分析和变换。

好的 AI IR 通常要表达：

- 张量 Shape 和类型。
- 算子语义。
- 控制流。
- 内存空间。
- 并行和布局信息。

## 12.11 AI 系统

AI 系统关注模型从训练到部署的完整工程，包括：

- 数据系统。
- 训练框架。
- 编译器和算子。
- 分布式执行。
- Checkpoint。
- 推理引擎。
- 服务调度。
- 监控和容错。

AI 算法回答“模型怎么算”；AI 系统回答“如何在真实硬件和业务约束下可靠、高效地算”。

## 12.12 训练系统

训练系统负责：

- 数据读取和 Shuffle。
- 前向、反向和参数更新。
- 混合精度。
- 分布式并行。
- Checkpoint。
- 容错恢复。
- 实验追踪和资源调度。

关键指标是训练吞吐、扩展效率、收敛时间、资源利用率和稳定性。

## 12.13 推理系统

推理系统负责：

- 模型加载。
- 权重和 KV Cache 内存管理。
- 请求排队。
- 动态批处理。
- Prefill/Decode 调度。
- 多卡并行。
- 流式输出。
- 故障隔离和监控。

指标包括 TTFT、TPOT、吞吐、P99 时延、显存利用率和单位 Token 成本。

## 12.14 端到端

端到端表示从真实输入进入系统，到最终输出产生的完整路径。它提醒工程师不要只优化局部：

- Kernel 快了，但编译时间可能变长。
- 融合减少 Launch，但可能降低并发。
- 大 Batch 提高吞吐，但可能违反时延 SLA。
- 压缩显存可能引入反量化开销。

最终优化目标必须与产品指标一致。

# 13. 分布式计算与存储系统

## 13.1 分布式计算

分布式计算让多个节点协作完成一个任务。大模型使用它是因为单设备的计算、内存和带宽不够。

核心问题：

- 数据和计算如何切分。
- 节点如何通信。
- 如何同步。
- 如何处理慢节点和故障。
- 如何保持结果一致。

## 13.2 节点、进程、Rank、World Size

- **Node**：物理或虚拟机器。
- **Process**：参与分布式任务的进程。
- **Rank**：进程在通信组内的编号。
- **World Size**：通信组总进程数。
- **Local Rank**：进程在本节点内的编号，常用于选择设备。

## 13.3 通信拓扑

拓扑描述设备之间的连接：

- 节点内互联通常带宽高、延迟低。
- 节点间通过网络连接，成本更高。
- Ring、Tree、Mesh 等拓扑适合不同消息大小和集合通信。

并行策略应尽量让通信密集的张量并行位于高带宽域内，把数据并行放在跨节点层次。

## 13.4 带宽与延迟模型

通信时间常粗略表示为：

```text
T = α + message_size / bandwidth
```

- `α` 是启动延迟。
- 大消息更受带宽影响。
- 小消息更受启动延迟影响。

因此过多小通信应合并；大通信应分块并与计算重叠。

## 13.5 扩展效率

从 1 张卡扩展到 N 张卡：

```text
speedup = T1 / TN
scaling_efficiency = speedup / N
```

效率下降原因：

- 通信增加。
- 负载不均。
- 数据输入不足。
- 串行部分。
- 同步等待。

## 13.6 Amdahl 定律

若程序可并行部分为 `p`，使用 N 个处理单元的最大加速比：

```text
S(N) = 1 / ((1-p) + p/N)
```

少量串行部分会限制无限扩展，说明优化端到端串行瓶颈的重要性。

## 13.7 分布式存储系统

分布式存储把数据分散在多节点，同时提供：

- 容量扩展。
- 并发访问。
- 数据复制或纠删码。
- 故障恢复。
- 一致性和元数据管理。

大模型训练中的数据集、Checkpoint 和模型权重通常依赖分布式存储。

## 13.8 Checkpoint

Checkpoint 是训练状态快照，可能包括：

- 模型参数。
- 优化器状态。
- 学习率调度器状态。
- 随机数状态。
- 数据迭代位置。

大规模训练中 Checkpoint 非常大，需考虑分片写入、异步保存、增量保存和故障恢复时间。

## 13.9 数据并行 I/O

如果所有卡同时读取相同存储，可能形成 I/O 热点。优化方式：

- 数据分片。
- 本地缓存。
- 预取。
- 多进程 DataLoader。
- 顺序大块读取。
- 数据打包。

## 13.10 一致性

一致性定义多个节点看到的数据状态是否一致。存储系统可能在强一致性、可用性和延迟之间权衡。

训练 Checkpoint 需要保证元数据与各分片属于同一逻辑版本，否则恢复时会混合不同 Step。

## 13.11 容错

容错包括：

- 节点故障检测。
- 任务重试。
- 从 Checkpoint 恢复。
- 弹性伸缩。
- 通信超时处理。

千卡训练中单机故障概率被规模放大，容错直接影响有效训练时间。

# 14. 并行软件开发

## 14.1 并行与并发

- **并发**：多个任务在同一时间段推进，不一定真正同时执行。
- **并行**：多个任务在同一时刻实际执行。

多线程可以提供并发；只有多核或多计算单元执行时才形成硬件并行。

## 14.2 进程与线程

- 进程有独立地址空间，隔离强，通信成本较高。
- 同进程线程共享代码、堆和全局数据，各自拥有栈、寄存器和调度状态。

线程切换通常比进程轻，但共享内存引入竞争和同步问题。

## 14.3 SIMD

SIMD 是 Single Instruction Multiple Data，一条指令同时处理多个数据元素，适合向量加法、激活、归约等规则计算。

高性能条件：

- 连续对齐数据。
- 分支少。
- 向量宽度充分利用。
- 尾部妥善处理。

## 14.4 SIMT

SIMT 是 Single Instruction Multiple Threads。大量线程按组执行相同指令，但每个线程有自己的寄存器和逻辑 ID。分支不一致会造成 Divergence。

## 14.5 数据并行与任务并行

- **数据并行**：相同程序处理不同数据分片。
- **任务并行**：不同任务或流水阶段同时执行。

算子 Kernel 多使用数据并行；训练系统同时结合数据、模型和流水并行。

## 14.6 竞争条件

多个执行单元并发访问共享数据，结果依赖执行顺序，称为 Race Condition。

解决：

- 避免共享可变状态。
- 原子操作。
- 锁。
- Barrier。
- 数据分区。
- 归约。

## 14.7 原子操作

原子操作在观察上不可分割。例如原子加可用于并发计数，但大量线程竞争同一地址会串行化，影响性能。

## 14.8 锁

- **互斥锁**：同一时刻只允许一个线程进入临界区。
- **读写锁**：多个读者可并行，写者独占。
- **自旋锁**：等待时循环检查，适合临界区短的场景。
- **乐观锁**：假设冲突少，通过版本/CAS 检测。
- **悲观锁**：先加锁再访问。

## 14.9 CAS

Compare-And-Swap 比较内存值是否等于期望值，若相等则原子替换。它是无锁数据结构的重要基础。

ABA 问题：值从 A 变 B 又变 A，CAS 看起来未变化，但中间状态已改变。可用版本号或标记指针解决。

## 14.10 Barrier

Barrier 要求所有参与者到达同步点后才能继续。同步过多会产生等待空洞，因此应尽量减少全局 Barrier。

## 14.11 死锁、活锁、饥饿

- **死锁**：任务循环等待彼此资源，永久阻塞。
- **活锁**：任务不断响应彼此但没有实际进展。
- **饥饿**：某任务长期拿不到资源。

预防死锁可统一锁顺序、减少嵌套锁、设置超时。

## 14.12 负载均衡

静态平均分块并不一定负载均衡，因为每块计算量可能不同。优化：

- 更细粒度任务。
- 动态任务队列。
- Work Stealing。
- 根据历史代价加权切分。
- 对尾块和稀疏度单独调度。

## 14.13 Work Stealing

每个工作线程有自己的任务队列；空闲线程从忙线程队列“偷”任务。它适用于任务耗时不规则的并行程序。

## 14.14 False Sharing

不同线程修改不同变量，但变量落在同一 Cache Line，导致 Cache Line 在核间频繁失效。可通过 Padding、对齐和数据分区解决。

## 14.15 内存一致性与可见性

编译器和 CPU 可能重排指令，各核 Cache 使一个线程的写入不一定立即被另一个线程看到。锁、原子操作和内存屏障用于建立正确的 Happens-before 关系。

# 15. 代码编写与算法实现能力

## 15.1 代码编写能力

岗位所说“较强代码能力”不仅是刷题，还包括：

- 正确表达算法。
- 处理边界和错误。
- 清晰的数据结构和接口。
- 可测试、可调试。
- 性能和内存意识。
- 读懂大型代码库。
- 使用版本控制和 Review。

## 15.2 算法实现能力

算法实现能力是把数学或论文描述变成正确、高效程序，包括：

1. 明确输入、输出和约束。
2. 选择数据结构。
3. 推导复杂度。
4. 证明或解释正确性。
5. 处理数值和边界。
6. 编写测试。
7. 根据 Profile 优化。

## 15.3 时间复杂度

时间复杂度描述输入规模增长时，操作数量的渐近增长：

- `O(1)` 常数。
- `O(log n)` 二分。
- `O(n)` 线性扫描。
- `O(n log n)` 高效比较排序。
- `O(n²)` 双重枚举。

它忽略常数和硬件，但用于排除不可扩展方案。

## 15.4 空间复杂度

空间复杂度描述额外内存随输入增长的关系。工程中还应关注：

- 峰值内存。
- 内存带宽。
- 分配次数。
- 碎片。
- Cache 局部性。

## 15.5 数据结构

该岗位常见：

- 数组与向量。
- 链表。
- 栈和队列。
- 哈希表。
- 堆 / 优先队列。
- 树和 Trie。
- 图。
- 并查集。
- LRU Cache。

## 15.6 算法范式

- 分治。
- 动态规划。
- 贪心。
- 回溯。
- 二分。
- 滑动窗口。
- 双指针。
- 图搜索。
- Top-K。
- 前缀和。

## 15.7 C++

C++ 在算子和系统开发中常用于：

- 高性能 Kernel Host 代码。
- 框架后端和运行时。
- 内存管理。
- 并发和模板库。

重点知识：RAII、指针与引用、对象生命周期、移动语义、智能指针、STL、并发、编译链接和性能。

## 15.8 Python

Python 常用于：

- 模型和算子原型。
- AI 框架前端。
- 测试和数据生成。
- 自动调优脚本。
- Agent 和工具编排。

性能敏感计算通常下沉到 C++/设备 Kernel；Python 更适合表达控制和实验逻辑。

## 15.9 编译与链接

典型流程：

```text
源代码 -> 预处理 -> 编译 -> 汇编 -> 目标文件 -> 链接 -> 可执行文件/库
```

- **静态库**：链接时复制需要的代码进入产物。
- **动态库**：运行时加载和共享。
- **JIT**：运行时根据 Shape 和硬件编译。

## 15.10 调试

高效调试流程：

1. 稳定复现。
2. 缩小输入。
3. 二分定位。
4. 检查不变量。
5. 添加断言和日志。
6. 与参考实现逐层比较。
7. 修复后添加回归测试。

# 16. 大模型训推系统实践、算子开发实践与工具

## 16.1 “具有实践经验”是什么意思？

不是“看过概念”，而是至少完成过闭环：

- 能运行真实模型。
- 能采集 Profile。
- 能找到一个热点或错误。
- 能修改实现或配置。
- 能用数据证明结果。
- 能解释失败方案和边界。

## 16.2 训练实践应掌握什么？

- 数据与 Tokenizer。
- Forward/Backward。
- Optimizer。
- 混合精度。
- 梯度累积和裁剪。
- 分布式训练。
- Checkpoint。
- 收敛和性能监控。

## 16.3 推理实践应掌握什么？

- 模型导出和加载。
- 权重格式。
- Prefill/Decode。
- KV Cache。
- Batch 和调度。
- 量化。
- 多卡并行。
- TTFT、TPOT 和吞吐测量。

## 16.4 算子开发实践应掌握什么？

- 参考实现。
- Shape/类型推导。
- Kernel 和 Host。
- Tiling。
- 多核和流水。
- 正确性测试。
- 精度分析。
- Profiling 和性能回归。

## 16.5 Profiling 工具

Profiling 工具用于采集 Timeline、Kernel 时延、硬件计数器、内存和通信信息。

可能接触的类别：

- 昇腾/CANN 相关 Profiler。
- PyTorch Profiler。
- GPU Profiler。
- CPU Profiler，如采样器、VTune、Perf。
- 分布式通信 Trace。

工具名会变化，面试重点是你会解释采集到的数据并形成优化闭环。

## 16.6 Trace

Trace 是按时间轴记录 Host、设备、Kernel、通信和同步事件。可观察：

- CPU 是否及时提交任务。
- Kernel 之间是否有空洞。
- Copy 与 Compute 是否重叠。
- 通信是否阻塞计算。
- 是否存在隐式同步。

## 16.7 Hardware Counter

硬件计数器记录：

- 指令数量。
- 计算单元忙碌周期。
- 内存事务。
- Cache 命中。
- 分支。
- Stall 原因。

计数器必须结合硬件和工作量解释，不能看到“利用率低”就直接下结论。

## 16.8 Tiling 策略

Tiling 策略是针对不同 Shape 和硬件选择：

- 切分维度。
- Tile 大小。
- 核数。
- 数据布局。
- Pipeline Stage。
- 尾块处理。

常见策略：

- 静态规则。
- Shape 分桶。
- 多版本 Kernel。
- Autotuning。
- 代价模型预测。

## 16.9 性能回归

性能回归测试要防止新改动使历史场景变慢。应维护：

- 关键 Shape 集。
- 不同数据类型。
- 时延阈值。
- 编译时间。
- 显存。
- 精度。
- 版本和环境信息。

## 16.10 Benchmark

Benchmark 是标准化比较方案。高质量 Benchmark 应：

- 代表真实工作负载。
- 公开输入与统计方式。
- 分离冷启动与稳态。
- 同时报告精度和性能。
- 避免只选择对自己有利的 Shape。

# 17. 竞赛与顶级会议

## 17.1 实习经验

实习经验的价值不只是“进过大厂”，而是证明你经历过真实工程约束：

- 代码需要被别人维护。
- 需求和时间有限。
- 结果要进入产品或研究流程。
- 需要 Review、测试、文档和协作。
- 性能数据必须可复现。

面试介绍实习时使用“背景—职责—难点—行动—结果—复盘”，不要只罗列工具。

## 17.2 高水平计算机竞赛

计算机竞赛通常考察：

- 算法设计。
- 数据结构。
- 编码速度。
- 正确性和边界。
- 团队协作。

它能证明算法与代码基础，但不能替代大型系统工程经验。

## 17.3 ACM/ICPC

ICPC 是 International Collegiate Programming Contest，历史上常称 ACM-ICPC。

典型形式：

- 三人一队。
- 共用一台计算机。
- 在限定时间内解决多道算法题。
- 按解题数和罚时排名。

常见知识：

- 图论。
- 动态规划。
- 数论。
- 字符串。
- 计算几何。
- 数据结构。

岗位为什么看重 ICPC：它能反映候选人的算法抽象、快速实现、Bug 排查和压力下协作能力。

## 17.4 数学竞赛

大学数学竞赛通常考察微积分、线性代数、概率统计等；数学建模竞赛还考察问题抽象、模型建立、数值求解、实验和论文写作。

与算子岗位相关的能力：

- 推导公式。
- 理解数值误差。
- 建立复杂度模型。
- 将数学过程实现为程序。

## 17.5 NeurIPS

NeurIPS 全称 Conference on Neural Information Processing Systems，是机器学习和人工智能领域的重要国际会议。

常见方向：

- 深度学习。
- 优化。
- 生成模型。
- 强化学习。
- 概率方法。
- AI 系统与社会影响。

如果简历写 NeurIPS 论文，必须能回答问题定义、方法、创新、实验、消融和个人贡献。

## 17.6 ICML

ICML 全称 International Conference on Machine Learning，是机器学习领域的重要国际会议。

研究覆盖学习理论、优化、生成模型、强化学习、表示学习和系统等。与 NeurIPS 有大量交叉，但会议历史、社区和每年主题侧重会有所不同。

## 17.7 AAAI

AAAI Conference on Artificial Intelligence 是人工智能领域的重要综合性国际会议，覆盖机器学习、知识表示、搜索、规划、自然语言、视觉、多智能体等。

## 17.8 VLDB

VLDB 通常指 Very Large Data Bases 领域及其旗舰会议 PVLDB/VLDB。它重点关注：

- 数据库系统。
- 查询优化。
- 分布式数据处理。
- 存储引擎。
- 流处理。
- 数据管理与 AI 系统交叉。

岗位列出 VLDB，说明团队不仅关注模型算法，也重视大规模数据、存储和系统能力。

## 17.9 “发表过文章”意味着什么？

面试官会追问：

- 论文解决什么问题？
- 为什么现有方法不够？
- 核心方法和公式是什么？
- 实验是否公平？
- 做了哪些消融？
- 失败案例是什么？
- 你个人完成了什么？
- 能否复现？

作者身份本身不是终点，真正看的是研究能力和可信贡献。

# 18. 技术调研、专利、报告与开源

## 18.1 业界前沿技术调研

技术调研不是搜索几篇博客，而是形成可决策的结论。建议结构：

1. 明确问题和评价指标。
2. 检索论文、官方文档、代码和 Benchmark。
3. 按技术路线分类。
4. 复现关键方案。
5. 在目标硬件和真实 Shape 上比较。
6. 分析适用条件、风险和成本。
7. 给出建议与下一步实验。

## 18.2 Literature Review

文献综述需要回答：

- 该方向如何演进？
- 主流技术路线是什么？
- 各自解决什么瓶颈？
- 假设和局限是什么？
- 哪些结论已被重复验证？
- 当前空白在哪里？

## 18.3 专利

专利是以公开技术方案换取一定期限排他权的制度。技术岗位常涉及发明专利。

算子领域可申请的内容可能包括：

- 新的数据流或 Tiling 方法。
- 新的融合方案。
- 动态 Shape 调度。
- 自动调优或自动代码生成。
- 精度补偿方法。
- 分布式通信与计算重叠。

## 18.4 专利申请

常见工作流程：

1. 发明挖掘。
2. 检索现有技术。
3. 明确技术问题、方案和效果。
4. 撰写交底书。
5. 与专利工程师完善权利要求。
6. 提交与审查答复。

技术人员要清楚：

- **新颖性**：现有技术没有公开相同方案。
- **创造性**：相对现有技术不是显而易见的简单组合。
- **可实施性**：方案能够被实现，而不是只有目标。

不要在专利提交前随意公开关键方案；具体流程应遵守公司知识产权制度。

## 18.5 技术交底书

技术交底书通常包括：

- 背景与现有问题。
- 发明目标。
- 完整方案。
- 流程图和结构图。
- 关键参数与实施例。
- 与现有方案的区别。
- 可量化效果。
- 可替代实现。

## 18.6 技术报告

技术报告服务于工程决策。常见结构：

1. 摘要和结论。
2. 背景与目标。
3. 环境和方法。
4. 正确性结果。
5. 性能结果。
6. 瓶颈分析。
7. 风险和限制。
8. 建议与后续计划。

好的报告应让他人能够复现实验。

## 18.7 Benchmark 报告

必须写清：

- 硬件、驱动、CANN/框架版本。
- Shape、DType、Layout。
- Batch 和序列长度。
- 预热与重复次数。
- 计时边界。
- 基线版本。
- 精度阈值。
- 原始数据与统计方法。

## 18.8 开源

开源是以特定许可证公开源代码，让社区使用、修改和贡献。

开源不等于“代码在网上可见”，还包括：

- 许可证。
- 贡献规范。
- Issue 和 PR 流程。
- 版本和发布。
- 测试与文档。
- 社区治理。

## 18.9 开源社区贡献

贡献形式：

- 修复 Bug。
- 新增算子或后端。
- 性能优化。
- 增加测试。
- 改进文档。
- 复现和报告问题。
- Review 他人的 PR。

高质量贡献应包含：

- 清晰问题描述。
- 最小复现。
- 正确性测试。
- 性能数据。
- 对兼容性的说明。

## 18.10 Pull Request

PR 是把代码修改提交给项目 Review 的工作单元。一个好的性能 PR 应说明：

- 原问题。
- 设计和替代方案。
- 为什么正确。
- Benchmark。
- 精度和回归测试。
- 影响范围。

## 18.11 技术影响力

技术影响力不是粉丝数，而是技术成果被采用和复用的程度，例如：

- 算子进入产品或主干库。
- 优化覆盖多个模型。
- 开源 PR 被合入。
- 文档成为团队标准。
- 专利、论文或报告推动技术决策。
- 培养他人并形成工程方法。

## 18.12 成果落地产品

“落地”表示成果从实验进入真实产品链路：

- 接入框架和发布流程。
- 覆盖真实模型与 Shape。
- 通过精度、稳定性和安全测试。
- 有监控、回滚和维护方案。
- 对用户或业务指标产生持续价值。

## 18.13 商业价值

算子优化的商业价值可体现为：

- 减少训练卡时和成本。
- 提高单机吞吐，减少服务器数量。
- 降低推理时延，改善用户体验。
- 支持更长上下文或更大模型。
- 降低能耗。
- 缩短模型上线周期。
- 提高昇腾生态的软件兼容性。

回答时尽量把技术指标转为业务指标：

> 单请求时延下降 20% 只是技术结果；在目标流量下减少多少实例、节约多少卡时、提升多少并发，才是商业价值。

## 18.14 高性能算子库

高性能算子库是经过大量 Shape、DType 和硬件优化的 Kernel 集合，通常包含：

- 算子 API。
- 多实现版本。
- Dispatch。
- Autotuning。
- Workspace 管理。
- 精度和性能回归。
- 框架适配。

## 18.15 基础加速库

基础加速库位于更底层，提供可复用能力，例如：

- BLAS/GEMM。
- FFT。
- 向量数学函数。
- 通信原语。
- 内存和线程原语。
- 排序、归约和 Scan。

上层算子库可组合这些基础能力。

## 18.16 算子开发框架

算子开发框架为开发者提供：

- 算子定义和注册。
- Shape/类型推导。
- Host/Kernel 构建。
- 编译。
- 测试和仿真。
- 调试。
- Profiling。
- 发布和框架集成。

它解决的是“如何标准化地开发大量可维护算子”，而不是只提供某个 Kernel。

# 19. 学习能力、自驱、团队意识与责任意识

## 19.1 学习能力

学习能力体现在：

- 能快速建立知识地图。
- 能阅读官方文档和源码。
- 能设计小实验验证理解。
- 能把知识用于真实问题。
- 能总结为可复用文档。

面试不要只说“我学得快”，要给具体时间线和成果。

## 19.2 自驱意识

自驱是没有人逐步安排时，仍能围绕目标推进：

- 主动发现问题。
- 设定可衡量目标。
- 拆解任务。
- 主动寻求资源。
- 持续汇报风险。
- 完成闭环。

自驱不等于不沟通或擅自扩大范围。

## 19.3 团队意识

团队意识包括：

- 共享信息。
- 尊重接口和约定。
- 及时暴露风险。
- 接受 Review。
- 帮助他人。
- 以团队目标而非个人炫技为优先。

## 19.4 责任意识

责任意识是对结果、质量和风险负责：

- 不隐瞒失败数据。
- 不为了性能破坏精度。
- 发现回归及时处理。
- 关键结论可复现。
- 对上线和维护有预案。

## 19.5 快速适应技术迭代

可用方法：

1. 跟踪官方 Release Note 和设计文档。
2. 建立最小可运行样例。
3. 用旧版本基线比较。
4. 识别 API、性能和精度变化。
5. 更新测试与文档。

## 19.6 责任分工

算子项目常涉及模型、框架、编译器、Kernel、硬件、测试多个角色。应明确：

- Owner。
- 输入输出接口。
- 验收标准。
- 时间节点。
- 风险和依赖。

## 19.7 冲突处理

推荐过程：

1. 先确认共同目标。
2. 区分事实、假设和偏好。
3. 用数据或小实验验证。
4. 记录决策和风险。
5. 无法达成时及时升级，而不是长期争论。
